# 0LUKA Artifact Model

Version: `v1.0`  
Status: `Artifact Provenance Specification`

## 1. Purpose

This document defines the artifact system of the 0luka platform.

Artifacts represent the observable outputs of runtime execution.

The artifact model ensures:
- traceability
- immutability
- runtime consistency
- operator visibility

## 2. Artifact Definition

An artifact is any persistent output generated by a job handler.

Examples:
- BOQ files
- cost estimates
- purchase orders
- reports
- analysis documents

Artifacts are referenced through:

`artifact_refs`

## 3. Artifact Lifecycle

Artifact creation flow:

```text
job handler
 → artifact generation
 → runtime sidecar record
 → artifact storage
 → projection
 → Mission Control
```

Artifacts must always originate from a handler execution.

## 4. Artifact Reference Structure

Example artifact reference:

```json
{
  "artifact_type": "boq_report",
  "path": "artifacts/qs/run_20260309_001/boq_report.json",
  "created_at": "2026-03-09T03:00:00Z"
}
```

Required fields:
- `artifact_type`
- `path`
- `created_at`

Optional fields:
- `checksum`
- `metadata`

## 5. Artifact Storage

Artifacts reside in runtime storage.

Example:

`runtime_root/artifacts/`

Typical structure:

```text
runtime_root/artifacts/
   qs/
      run_20260309_001/
         boq_report.json
         cost_estimate.json
```

Artifacts are immutable once created.

## 6. Artifact Truth Path

Artifact references propagate through the runtime system.

```text
handler
 → runtime state sidecar
 → outbox projection
 → Mission Control
```

The runtime sidecar is the source of truth.

## 7. Artifact Integrity Rules

Artifacts must satisfy the following rules.

Deterministic Generation:
- artifacts must be produced by a handler execution.

Immutability:
- artifacts must never be modified after creation.

Provenance:
- every artifact must map to:
- `run_id`
- `job_type`
- `handler`
- `timestamp`

## 8. Failure Behavior

If artifact generation fails:

`run state → FAILED`  
`artifact_refs → []`

Artifacts must never be fabricated.

## 9. Artifact Validation

The runtime validator may check:
- file existence
- reference integrity
- path validity

Missing artifacts trigger:

`ARTIFACT_ERROR`

## 10. Artifact Projection

Artifacts appear in operator interfaces via projection.

Example API:
- `/api/qs_runs`
- `/api/qs_runs/{run_id}`

These endpoints expose artifact references.

## 11. Artifact Indexing (Future)

Future systems may add:
- artifact search
- content indexing
- analytics
- cross-run artifact comparison

These features must never replace the runtime sidecar as the truth source.

## 12. Artifact Security

Artifacts must respect platform access rules.

Sensitive artifacts may require:
- approval
- restricted visibility
- operator permission

## Summary

Artifacts represent the persistent outputs of runtime execution.

The artifact model ensures:
- traceable outputs
- immutable records
- consistent runtime state
- operator visibility

Artifacts are the observable evidence of system execution.
